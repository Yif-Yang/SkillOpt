Anatomy of a Validation Gate: What 280+ Runs Taught Us About Training Agent Skills

SkillOpt trains an agent's skill document the way you train weights — with epochs, batches, and a held-out validation gate. But when does the gate actually matter? Should the epoch-end slow update be gated too? Can reflection be made skill-aware? Can you skip the gate entirely? We ran a 280+-run ablation campaign to find out — and shipped the answers as SkillOpt-Sleep, a nightly self-evolution plugin for Claude Code, Codex, and Copilot.

The campaign

SkillOpt's core loop: a frozen target agent rolls out tasks; a separate optimizer model distills bounded add / delete / replace edits to a single skill document; a candidate is accepted only if it does not lose to the current best on a held-out validation slice. That last step — the validation gate — is the design choice this campaign interrogates.

We crossed five gate arms with four optimization variants (step-size × schedule: def / lr8 / const / lr8const) on three benchmarks with real, execution-grade evaluators, across four target models (GPT-5.5, GPT-5.4-mini, GPT-5.4-nano, and a local Qwen3.5-4B). Optimizer: GPT-5.5 throughout.

ArmGateMeaning
C1offEvery candidate skill is accepted unconditionally (ablation control; validation still measured).
C2hardAccept only if mean exact-correctness on the val slice does not drop.
C3softGate on partial-credit score (token-F1 / graded points / per-case progress).
C4mixedGate on (hard + soft) / 2.
C5hard + gated slow-updateLike C2, but the epoch-end slow update must also pass validation selection instead of being force-injected.

Benchmarks: SearchQA (1,400-item held-out test, SQuAD exact match), LiveMathematicianBench (124 frontier-math MCQs), and SpreadsheetBench (280 cases where the agent's generated openpyxl code is actually executed and the produced workbook is compared cell-by-cell against a golden one).

Methodology note, stated up front: each (arm × benchmark × model) cell aggregates 4 hyper-parameter variants at a single seed, so we treat differences under ~1 point as noise, tier our findings by confidence, and have publicly retracted one early finding that didn't survive de-pooling. The full per-run tables — every cell, including the unflattering ones — are committed alongside this post.

Question 1

When does the validation gate matter — and which metric should it use?

Answer: it depends on the task's signal quality — and once the task has partial credit, the gate's metric is worth real points.

Net lift by gate arm (final skill vs baseline, mean over 4 variants × 4 models)

BenchmarkC1 no gateC2 hardC3 softC4 mixed
SearchQA+9.9 pts+10.0 pts+11.2 pts+11.0 pts
LiveMathematicianBench+12.5 pts+13.2 ptsn/a (MCQ has no partial credit)
SpreadsheetBench+18.9 pts+19.7 pts+24.0 pts+26.0 pts
Finding 1 — On clean-signal tasks the gate is insurance, not fuel. On SearchQA every arm lands within a point of +10: turning the gate off, or switching its metric, doesn't change the destination. The optimization signal is so clean that almost every proposed edit is genuinely good — the gate simply has nothing to reject.
Finding 2 — On partial-credit tasks, the gate metric is worth up to +7 points. SpreadsheetBench scores progress within each case. Gating on that soft / mixed signal (C3/C4) beats both no-gate and hard-gate by 4–7 points on average — the densest-information signal makes the best accept/reject decisions. If your task has partial credit, gate on it.
Finding 3 — Where self-evolution actually goes wrong: weak model × noisy task. The one place we measured negative optimization at research scale is LiveMath × GPT-5.4-nano: in 4 of 8 runs the "optimized" skill is worse than the initial one (e.g. 0.282 → 0.242). A weak model on a noisy, no-partial-credit task ends up fitting noise. Remember this cell — it reappears, much more dramatically, in the deployment section below.

The ranking that emerges: task signal quality ≫ model tier > hyper-parameters ≈ gate switch — but the gate's value is asymmetric: it costs nothing where it's redundant and pays off exactly where things get risky.

Question 2

Should the epoch-end slow update be gated too?

SkillOpt periodically consolidates the epoch's lessons into the skill — the slow update. C1–C4 force-inject it. C5 makes it pass validation selection like any other candidate. The result is one of our favorite findings of the campaign.

C2 (slow update force-injected)C5 (slow update validation-gated)
Mean net lift (46 paired runs)baseline−0.3 pts (statistically indistinguishable)
Runs where final skill = best skill10 / 4740 / 47
Gating the slow update doesn't buy points — it buys determinism. Performance is a wash, but under C5 the artifact you ship at the end of training is the artifact that validated best in 40 of 47 runs (vs 10 of 47 with force-injection). No more "the best checkpoint was actually three steps ago": the final skill and the best skill become the same file. For anything that deploys automatically — like the nightly plugin below — that property is worth more than a point of accuracy.
Question 3

Skill-aware reflection: don't delete rules that weren't the problem

A failure doesn't always mean the skill is wrong. Sometimes the rule was right and the model just didn't follow it. Standard reflection can't tell the difference — so it "fixes" valid rules and erases hard-won knowledge.

Skill-aware reflection (SAR) makes the optimizer triage every failure first: SKILL_DEFECT (the rule is wrong → edit the skill body) vs EXECUTION_LAPSE (the rule is fine but was ignored → leave the body alone and strengthen a protected appendix instead). We tested two routing modes against the matched C1/C2 baselines on GPT-5.5, same hyper-parameters, same seeds:

SAR modePaired runsMean Δ lift vs baselineWins
Route successes + failures19+0.3 pts10/19
Failure-only routing22+1.6 pts13/22
Finding — triage failures, leave successes alone. Routing only failures through the defect/lapse split is the configuration that helps (+1.6 points on average over 22 paired runs, on top of an already-optimized baseline). The mechanism matches the intuition: most of SAR's value comes from not destroying valid rules, and successes were never going to destroy anything. Several SAR cells set the campaign's per-benchmark records — e.g. SearchQA GPT-5.5 0.776 → 0.876, the best SearchQA score of the entire campaign.
Question 4

Can we skip the gate? Post-hoc selection and skill-feature-side checks

Validation rollouts are the gate's main cost. Two cheaper alternatives — one validated by the campaign, one earlier-stage but already useful in production.

Validated: train gate-free, select post-hoc

Because C1 still measures validation each step (it just doesn't reject), we can compare gate-free training + after-the-fact checkpoint selection against true in-training gating. On clean-signal tasks they are equivalent — C1's post-hoc-selected checkpoint matches C2's gated result within noise. And a related, deliberately deflating observation: the last checkpoint is not worse than the val-best checkpoint in most runs (under a hard gate, 13/16 ties-or-wins). The val signal is a guardrail, not an oracle — don't over-trust per-step val rankings, and don't pay for more validation than the task's noise level requires.

Earlier-stage: selection by skill-side features, no rollouts at all

The longer-term question is whether a candidate edit can be screened by looking at the skill itself — its constraints, structure, and consistency with the task's contract — without spending a single validation rollout. Our first production instance of this idea is the output-contract guardrail that ships in SkillOpt-Sleep: the optimizer is shown the task's hard constraints (required output format and language, no clarifying questions) and any proposed rule that violates them is rejected by construction — a pure feature-side check with zero rollout cost.

Measured effect of the feature-side guardrail (deployment runs, gate-free mode): on SpreadsheetBench it flipped gate-free consolidation from regressing to gaining on both the strongest and weakest targets we tested (GPT-5.5: −0.4 → +0.4 pts; GPT-5.4-nano: −5.0 → +2.9 pts) — by construction it can never reject a useful edit, only ill-formed ones. Richer feature-side critics (rule redundancy, scope, conflict detection) are the open research direction this campaign points to next.
From research to product

SkillOpt-Sleep: the recipe, shipped as a nightly plugin 😴

Everything above is distilled into an open-source plugin for Claude Code, Codex, and GitHub Copilot. While you sleep, your idle API quota turns into a five-stage offline cycle on your own usage history:

1HarvestCollect recent session transcripts from your local agent.
2MineDistill the recurring tasks you actually ask for.
3DreamReplay each task in multi-rollout groups + synthetic variants.
4ConsolidateDistill bounded skill edits from the good-vs-bad contrast.
5GateShip an edit only if it passes the held-out gate (Q1) — with the campaign's defaults baked in.

The campaign's answers are the plugin's defaults: validation gate on (Q1), gated consolidation of nightly updates (Q2), contrastive failure-focused reflection (Q3), and the feature-side guardrail protecting the gate-free mode (Q4). The output is a compact, human-readable skill block in your agent's memory — reviewable, revertible, zero inference-time overhead, and fully decoupled from the research codebase.

Deployment-scale validation

Research training uses hundreds of labeled tasks; a real user has ~10 a day. So we re-ran the protocol at deployment scale — 5 nights, 10 new real tasks per night, dream-augmented — against the same full test sets and evaluators as the campaign. End-to-end with real agents, on the public gbrain-evals skillopt-v1 benchmark, the plugins drive real Claude Code and real Codex through the full cycle: deficient seed skills go 0.00 → 1.00 on held-out tasks in all 4 seeds. Remember Finding 3? We stress-tested it.

Q1's riskiest cell — weak model, noisy self-evolution signal — is exactly what a no-gate nightly loop faces in the wild. So we ran the ablation: same weak model (GPT-5.4-nano, SearchQA), same nights, same candidate edits, gate off vs gate on.

This is the whole campaign in one picture. Ungated, the optimizer adopted a plausible-but-wrong rule and the obedient small model rode it off a cliff (−52.8 points in five nights). The gated twin rejected those same edits and never lost a point. Research said the gate is insurance that matters most for weak models on noisy signals — deployment proved it. SkillOpt-Sleep ships with the gate on by default.
