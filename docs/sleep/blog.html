The idea

Agents that sleep on it

Coding agents see the same kinds of tasks from you every day — and forget everything between sessions. Meanwhile your API quota sits idle all night. SkillOpt-Sleep connects the two.

Every night (scheduled, or on demand with /skillopt-sleep), the plugin runs a five-stage cycle, entirely offline — your agent is never interrupted, and nothing is deployed unless it provably helps:

1HarvestCollect recent session transcripts from your local agent history.
2MineDistill recurring tasks — the things you actually ask for, again and again.
3DreamReplay each task many times (multi-rollout groups) and augment with synthetic variants — cheap offline experience.
4ConsolidateA separate optimizer distills bounded skill edits from the good-vs-bad rollout contrast.
5GateAn edit ships only if it improves a held-out validation slice of your own real tasks.

The output is a compact, human-readable skill block appended to your agent's memory (CLAUDE.md / SKILL.md) — reviewable, revertible, and adding zero model calls at inference time. The tool is fully decoupled from the research codebase: skillopt_sleep/ has no dependency on the paper code.

The science

Built on SkillOpt: skill training that behaves like an optimizer

SkillOpt-Sleep is the deployment-time companion of SkillOpt, which treats the skill document as the trainable state of a frozen agent — with epochs, batches, a textual learning rate, and a validation gate.

At research scale (full training splits, 40 optimization steps), the latest ablation campaign — 192 gated-training runs across C1–C5 arms, plus a new skill-aware reflection (SAR) variant — shows large, reproducible held-out gains across every model tier:

Two findings from the ablations matter most for deployment:

  • The validation gate is what makes self-evolution dependable. Gated arms (C2–C5) deliver the same peak gains as the ungated arm while sharply limiting worst-case drift; the newest C5 arm (validation-selected slow updates) makes the final skill coincide with the best skill in the large majority of its 48 runs.
  • Weaker models gain the most. The lower the baseline, the more an optimized skill adds — nano and the local 4B model see the biggest jumps.
The plugin, validated

Deployment-scale: real gains from a handful of tasks per night

Research training uses hundreds of labeled tasks. A real user has maybe ten a day. So we re-ran the protocol at deployment scale — 5 nights, only 10 new real tasks per night, dream-augmented — against the same full held-out test sets and the same real evaluators as the research campaign.

It works end-to-end with real agents

On the public gbrain-evals skillopt-v1 benchmark, the shipped plugins drive real Claude Code and real Codex through the full sleep cycle: deficient seed skills go from 0.00 → 1.00 on held-out tasks in all 4 seeds, with Claude and Codex cross-verifying each other's consolidated skills.

Gains arrive fast — and the gate locks them in. The headline SearchQA run learned its winning rule on night 2 and held the improved score for every night after: 0.797 → 0.797 → 0.844 → 0.844 → 0.844 → 0.844. You don't need weeks of usage history — a couple of nights of normal use is enough for the sleep cycle to converge on its lift, and the validation gate prevents later edits from eroding it.

Every baseline above is genuine and matches the research campaign's numbers for the same model (e.g. SearchQA GPT-5.5 ≈ 0.79 in both), so the deployment results are directly comparable with the research tables — same data splits, same evaluators, same scoring.

The safety story

We stress-tested naive self-evolution so you don't have to

What happens if an agent just accepts every "lesson" it draws from its own replays — no validation, no gate? We ran exactly that ablation. The answer is the strongest argument for SkillOpt-Sleep's design.

On SearchQA with the weakest target (GPT-5.4-nano), the ungated ablation arm let the optimizer adopt a plausible-sounding but wrong rule. The obedient small model followed it, and accuracy collapsed night after night. The gated run — same model, same nights, same candidate edits — rejected every one of those edits and never lost a point:

Night 0 baseline
0.554
After 5 nights — no gate (ablation)
0.026
After 5 nights — with the gate (default)
0.570
This is the product thesis in one experiment. Self-evolving agents are easy to build and easy to ruin. The difference between a nightly loop that compounds value and one that quietly destroys your agent is a held-out validation gate — the same mechanism that makes SkillOpt's research training stable. SkillOpt-Sleep ships with the gate on by default, and our stress tests show it filtering harmful edits at every model tier.

And for users who can't hold out a validation set

The gate needs a slice of real tasks to validate against. For daily cases where that's impractical, SkillOpt-Sleep offers a gate-free mode hardened by an output-contract guardrail: the optimizer is shown the task's hard constraints (output format, language, no clarifying questions) and is forbidden from proposing rules that violate them. In our SpreadsheetBench runs this guardrail single-handedly turned the gate-free mode from regressing into the positive gains shown above — on both the strongest and weakest targets. Stronger targets are naturally robust in gate-free mode (GPT-5.5's drift across all our gate-free runs stayed within a few points); for smaller targets we recommend keeping the default gate on.

What's in the box

Features

Three agents, one cycle

First-class plugins for Claude Code, Codex CLI, and GitHub Copilot (MCP server) — same engine, shared skills.

Built-in scheduling

/skillopt-sleep schedule 02:30 — the cycle runs itself every night; unschedule to stop.

Validation gate (default on)

Hard / soft / mixed gate metrics, vendored self-contained from the SkillOpt research gate.

Gate-free mode + guardrail

For daily cases without a val set: greedy consolidation, protected by the output-contract guardrail.

Dream rollouts

Multi-rollout contrastive reflection — replay each task K times and learn from the good-vs-bad contrast.

User preferences & budgets

Steer the optimizer in plain text (accuracy / tokens / latency), cap nightly token spend and wall-clock.

Review-then-adopt

Every night writes a staging report; skills are plain markdown you can read, edit, or revert.

Decoupled & dependency-free

skillopt_sleep/ imports nothing from the paper code; the engine runs on stock Python.
